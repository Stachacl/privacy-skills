# Vendor Data Sharing Review

Review what data is being shared with a vendor, consultant, SaaS tool, or AI service before sending it, and check whether safer alternatives exist.

## When to use this

- You are about to send customer data, employee data, or internal documents to a vendor or external service.
- You want to review what a current vendor receives and whether it is still appropriate.
- You are onboarding a new SaaS tool and need to understand what data it will have access to.

## The checklist

1. Identify what type of data is being shared: customer data, employee data, financial data, internal documents.
2. Check why the vendor needs this specific data and whether a redacted or anonymised version would work.
3. Check whether synthetic or test data could be used instead.
4. Check who at the vendor can access the data.
5. Check where the data is stored and in which region.
6. Check how long the vendor retains the data.
7. Check whether the vendor uses subprocessors and who they are.
8. Check whether the vendor uses the data to train AI models.
9. Check whether a data processing agreement (DPA) or data sharing agreement is in place.
10. Check whether the contract limits how the vendor can use the data.
11. Consider safer alternatives: sharing on a need-to-know basis, using a secure transfer portal, or time-limiting access.

## Examples

- A customer support team sends a full customer export to a vendor for a one-off analysis. The export includes fields the vendor does not need.
- A marketing agency is given access to a live CRM with all customer records when they only need a filtered export for a specific campaign.
- An AI SaaS tool is integrated with customer data. The vendor terms allow using customer data to improve their model unless the enterprise plan is active.
